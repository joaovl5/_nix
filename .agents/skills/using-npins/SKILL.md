# Using npins

## Model

- `npins/sources.json` is the source of truth for pinned external inputs
- `npins/default.nix` is generated by npins; never edit it by hand
- `inputs.nix` adapts pins into the repo's flake-shaped `inputs` set
  - raw source pins stay as source paths/metadata
  - flake pins are loaded through `flake-compat`
  - `rev`/`shortRev` are normalized there for consumers
- `flake.nix` is only a thin shim over `default.nix`; never add inputs there

## Hard boundaries

- We **do not** want transitive dependencies in `npins/sources.json`
  - do not mirror upstream `inputs`, `follows`, lock graphs, or helper pins
  - do not add an upstream dependency just because a pinned flake depends on
    it
  - add a pin only when this repo imports, builds, updates, or patches it
    directly
- Let `flake-compat` consume an upstream flake's own lock when loading that
  flake
- Do **not** reintroduce `flake-file`, `unflake`, or `with-inputs`
- Do **not** hand-edit generated npins files

## Adding pins

- Prefer direct pins with clear names
  - package source-only repos usually use a `-src` suffix
  - flake-like dependencies keep their logical input name
- Use the npins CLI for source changes
  - add GitHub: `npins add github OWNER REPO --name NAME`
  - add Git URL: `npins add git URL --name NAME`
  - update: `npins update NAME`
- Prefer the existing mirror style when a mirror already exists in
  `npins/sources.json`
- Stage only `npins/sources.json` for pure pin changes unless adapter or
  package code also changed

## Wiring pins

- If a pin should behave as `inputs.<name>` flake outputs, leave it out of
  `raw_inputs` in `inputs.nix`
- Add a pin to `raw_inputs` only when repo code needs the fetched source path
  or metadata, not flake outputs
- Keep `inputs.nix` small and mechanical
  - import `./npins`
  - import `flake-compat`
  - choose raw vs. flake-loaded inputs
  - normalize `rev` and `shortRev`
- Keep local repo data local; never pin `globals/` or other in-tree paths

## Secrets pin

- For `mysecrets`, push the secrets repo first
- Then run `nix-shell -p npins --run 'npins update mysecrets'`
- Stage `npins/sources.json` after the update

## Checks

- Run `npins verify` when `npins/sources.json`, `npins/default.nix`, or
  `inputs.nix` changes
- Run targeted eval or build for the affected input/package/service
  - example: `nix eval --impure --json .#supportedSystems`
- Then run the repo-standard checks
  - `nix fmt`
  - stage exact intended files
  - `prek`
  - `nix flake check --all-systems` only when Nix code changed
    - if blocked locally, run `nix flake check`, relevant targeted checks, and
      document the blocker
