# Frag Optimization Plan

> **For agentic workers:** REQUIRED: Use `superpowers:subagent-driven-development` if subagents are available. Keep exactly one task in progress per worker. Treat this document as the execution source of truth. Do not commit unless the user explicitly asks.

**Goals:**

- materially reduce frag startup latency
- fix highest-risk structural problems in the frag codebase, using current `main` as the verified baseline, so future latency work is easier to implement, benchmark, and review.

**Architecture:** treat frag optimization as four connected workstreams:

1. measurement and test guardrails
2. shared runtime-contract cleanup
3. execution-worktree wins for image reuse / image size / recurring bootstrap cost against current-main bottlenecks
4. first-cold-start architecture experiments around image ingestion.

This plan is stored on `main` for coordination, but implementation should happen in a dedicated worktree based on current `main`. Do not conflate current `main` with the separate remediation worktree; use the remediation worktree as a source of candidate backports unless you intentionally choose it as the execution branch and capture a fresh baseline first.

**Tech Stack:**

- Python CLI/runtime (`_scripts/frag/src/frag`)
- Pytest
- Nix packaging (`packages/frag`)
- Docker/rootless Docker
- NixOS image building.

## 2. Verified current-main snapshot

These facts were re-verified before writing the plan.

### Startup path

Current cold `frag enter` on `main` still does:

1. resolve profile + runtime spec,
2. load profile image,
3. start container,
4. wait for bootstrap,
5. `docker exec` the requested command.

Primary files:

- `_scripts/frag/src/frag/cli.py`
- `_scripts/frag/src/frag/docker_runtime.py`
- `_scripts/frag/src/frag/image_assets.py`

### Current image-ingestion shape

Current `main` still:

- evaluates a NixOS system in `packages/frag/images.nix`
- builds a flat rootfs tar
- rewrites `/home/agent` and `/sw`
- then always runs:
  - `docker image rm -f "$image_ref" || true`
  - `docker import ... "$runtime_rootfs" "$image_ref"`

### Current size / latency picture

Retained from the verified startup-latency report:

- first cold start is still dominated by image ingestion/materialization
- repeat-start latency is dominated by bootstrap/lifecycle work once the image exists locally
- the runtime image is large, and current `main` still bakes in heavy packages

Measured suspects from the latest report:

- `llm-agents.agent-browser`: ~1.7 GiB
- `git`: ~347.7 MiB
- `llm-agents.omp`: ~330.8 MiB
- `mcp-nixos`: ~313.5 MiB
- `llm-agents.code`: ~296.8 MiB

## 4. Planning principles and sequencing rules

1. **Do not optimize against the wrong code state.**
   - Benchmark `main` separately from remediation-worktree behavior.
2. **Do not let cleanup swallow the performance work.**
   - Only do structural cleanup that reduces drift/risk for planned optimization steps.
3. **Update tests in lockstep with behavior changes.**
   - Current tests explicitly assert old packaging/runtime behavior.
4. **Keep repeat-start work and first-cold-start work conceptually separate.**
   - They touch different bottlenecks.
5. **Prefer invariant-based tests over implementation-shape tests.**
   - Example: verify image metadata correctness, not “must call docker import forever.”
6. **Use the remediation worktree selectively.**
   - Cherry-pick ideas and verification harnesses, but do not assume the worktree is already correct for current `main` without re-checking.

## 6. Success criteria

The optimization effort is successful only if all of these hold.

### Performance

- materially improved **first cold start** relative to current-main baseline
- preserved or improved **repeat non-running start** behavior
- no regression in already-good running-container reuse

### Correctness

- `/init`, `/sw`, `/home/agent`, and current runtime user semantics remain valid unless intentionally redesigned
- read-only shared assets remain enforced
- writable home persistence remains correct
- workspace ownership semantics remain correct
- failure paths do not leave orphaned containers/volumes behind

### Engineering quality

- current drift-prone contracts are centralized enough that follow-on work is safer
- tests no longer encode obsolete implementation details as hard requirements
- benchmarks clearly record code state + host/runtime context
