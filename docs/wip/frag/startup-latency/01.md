## Preliminary internal static findings

> These are static-analysis findings from parallel read-only code review. They are strong hypotheses, not runtime-profile measurements.

### Ranked likely contributors

1. **Bootstrap recursively repairs ownership of persistent state on cold start**
   - Current static evidence points to recursive `chown` work in `_scripts/frag/src/frag/bootstrap.py` as the biggest repeated cold-start cost.
   - This appears to scale with the size of the profile state/home tree.
   - Important nuance: this is likely **cold-start-per-container** cost, not just first-ever-host cost.

2. **Bootstrap readiness polling does multiple Docker/status probes per retry**
   - `_scripts/frag/src/frag/docker_runtime.py` currently does repeated readiness/status/log checks while waiting for bootstrap.
   - Static analysis suggests this multiplies Docker round trips during cold start and can add visible latency even when the container is otherwise healthy.

3. **Image helper work sits on the critical path before the shell is ready**
   - The `frag enter` cold path synchronously goes through packaged image/helper resolution before startup.
   - Even if the expensive part is first-host-start-only, this still means the helper path is in the critical path for cold starts.

4. **Current NixOS runtime image packaging uses a flat rootfs import flow**
   - Static packaging analysis indicates the runtime image path still depends on a full rootfs tar/import workflow.
   - This is likely the dominant **first-host-start / first-image-materialization** cost.

5. **Runtime system closure size likely amplifies first import cost**
   - The runtime image includes a real NixOS userland plus multiple agent tools.
   - Static analysis suggests this increases import/unpack time for the initial image materialization step.

### Preliminary split: one-time vs recurring

- **Likely one-time or cacheable per host/image ref**
  - image materialization/import of the NixOS-derived runtime image
  - closure/rootfs size impact during the first image ingest

- **Likely recurring on each non-reuse cold start**
  - bootstrap ownership repair / recursive `chown` work
  - readiness polling / repeated Docker status probes
  - other fixed startup orchestration overhead before `docker exec` returns a shell

### Preliminary remediation ideas from internal analysis

- avoid duplicate or unnecessary recursive ownership walks
- persist and compare last-applied ownership metadata so unchanged trees can skip repair
- reduce readiness polling round trips or use a more direct readiness contract
- avoid repeated helper work once an exact `image_ref` is already known locally
- reduce rootfs/system closure size where possible
- consider image-loading strategies that avoid full flat-rootfs import cost on first use

## External research update — Docker/rootless cold-start behavior

### What the completed Docker-focused research suggests

1. **`docker import` is likely a major first-use penalty in the current design**
   - Docker documents `docker image import` as creating an image from a tarball/rootfs stream, with only limited metadata restored via `--change`.
   - Docker documents `docker image load` as restoring an image/repository tar archive with images and tags preserved.
   - For `frag`, the current helper in `packages/frag/images.nix` still performs a rootfs-tar-based `docker import`, which likely means the full NixOS filesystem tar must be ingested before first use on a host.
   - This looks like a **one-time per host / per image-ref** cost, but likely a large one.

2. **The current helper behavior can make image materialization cost recur**
   - The completed research notes that the helper still removes/re-imports by exact `image_ref` instead of behaving like a cheap no-op once the exact image is already present.
   - If `frag enter` always goes through that helper path, then a cost that could be one-time may become part of repeated cold-start behavior.

3. **Rootless Docker storage-driver choice may strongly affect performance**
   - Docker rootless troubleshooting docs say supported drivers include `overlay2`, `fuse-overlayfs`, `btrfs`, and `vfs`, with `vfs` documented separately as much slower because it deep-copies layers instead of using union copy-on-write behavior.
   - If the local rootless Docker daemon is on `vfs`, cold image/container work can be much slower than expected.

4. **Docker Engine 29+ containerd image store may open mitigation options**
   - Docker documents that fresh Docker Engine 29 installs use the containerd image store by default.
   - Docker also notes that this enables advanced snapshotters such as lazy-pulling/image-distribution options.
   - For `frag`, this matters less as a direct fix right now, but it suggests there may be better image-materialization strategies than the current flat-rootfs import path.

5. **Rootless daemon startup itself can add first-command latency**
   - Docker rootless docs note lingering/user-daemon startup considerations.
   - This may add a smaller first-command cost on some hosts, though it looks secondary compared with image import and bootstrap work.

### Preliminary remediation ideas from the Docker-focused research

- **Highest-value likely fix:** stop relying on a flat-rootfs `docker import` path for routine local interactive startup if we can preserve the NixOS-derived runtime another way.
- If we keep the current helper shape, make it a true fast-path no-op when the exact `image_ref` already exists locally.
- Check what storage driver the rootless daemon is using; if it is `vfs`, that is a strong suspect for exaggerated cold-start cost.
- Investigate whether a `docker load`-style preserved image archive, or another Nix-built image format that avoids repeated rootfs import, can preserve the current design goals with lower first-use cost.

## External research update — Nix/NixOS image-build strategy

### What the completed Nix-focused research suggests

1. **The current `frag` image path is the upstream-documented NixOS rootfs import flow**
   - `packages/frag/runtime-system.nix` imports the NixOS Docker-image machinery.
   - Upstream nixpkgs documents this pattern as: build `config.system.build.tarball`, then `docker import` it and run `/init`.
   - So the current design is legitimate, but it inherits the cold-start tradeoff of a rootfs-import flow.

2. **That rootfs-import flow is different from `dockerTools` image archives**
   - Upstream `dockerTools` builders (`buildImage`, `buildLayeredImage`, `streamLayeredImage`) produce Docker-compatible repository/image archives rather than the raw-rootfs import path.
   - This matters because `docker load`/layer-preserving image archives are a materially different ingestion path from `docker import` of a flat rootfs tar.

3. **`streamLayeredImage` exists specifically to reduce IO and improve image-ingest workflows**
   - The nixpkgs work that introduced `streamLayeredImage` explicitly cites heavy IO from realizing tar layers/final archives and positions streaming as a speed/IO improvement for cases where images are immediately piped to Docker/skopeo.
   - This does not directly prove it is the right answer for `frag`, but it is strong upstream evidence that the Nix image-build choice can have major performance consequences.

4. **A simple builder swap is not obviously safe for `frag`**
   - The completed research explicitly notes an important caveat: preserving all current NixOS runtime invariants (`/init`, `/sw`, current identity/home assumptions, etc.) would take design work if we move away from the current rootfs-import path.
   - So this is not just “replace import with dockerTools and call it done.”

5. **There is still likely room for a hybrid mitigation strategy**
   - Even if we keep the NixOS-derived runtime concept, it may be possible to ship a prebuilt Docker/OCI archive shape that is faster to ingest than the current flat-rootfs import path.
   - Another possible direction is to keep the NixOS rootfs approach but reduce repeated helper/import work and reduce the runtime closure size so the one-time import penalty shrinks.

### Practical implications for `frag`

- **Likely biggest one-time cost:** current NixOS rootfs tar -> `docker import` path
- **Likely recurring costs:** bootstrap/setup work after the image exists locally
- **Most promising architecture-level mitigation:** investigate whether `frag` can keep a NixOS-derived runtime while shipping a loadable OCI/Docker archive shape instead of re-importing a flat rootfs tar
- **Most conservative near-term mitigation:** keep the current architecture but make image materialization strictly one-time per exact `image_ref`, and reduce runtime closure size/import frequency

### Important caveat from the research

The Nix-focused research also surfaced a correctness/design caveat for `dockerTools`-layered images with Nix store content: upstream docs/code warn that only the deepest Nix layer’s database is loaded cleanly. That does not automatically rule out `dockerTools` for `frag`, but it means any migration away from the current rootfs-import path needs careful design rather than an automatic builder swap.

## Current working hypothesis

If we combine:

- internal static analysis
- Docker/rootless external research
- Nix/NixOS image-strategy external research

then the current best hypothesis is:

1. **First-use startup latency is dominated by the current NixOS rootfs -> `docker import` image materialization path**, especially if the rootless daemon/storage setup is suboptimal.
2. **Repeated non-reuse cold starts are then dominated by bootstrap-side work** such as ownership repair and readiness/status polling.
3. **Best remediation directions split into two layers:**
   - **architecture-level:** reduce or replace the flat-rootfs import path while preserving NixOS runtime guarantees
   - **runtime-level:** reduce recurring bootstrap work, especially recursive ownership repair and polling overhead

## Next likely practical follow-up

A good next investigation would be a targeted design study of three concrete mitigation options:

1. keep current NixOS rootfs-import design, but make helper/import strictly one-time and aggressively reduce bootstrap overhead
2. preserve NixOS-derived runtime semantics while generating a Docker/OCI archive shape that can be `docker load`ed instead of rootfs-imported
3. slim the runtime closure and/or split rarely used tools so first import cost drops materially
