# Frag startup latency broader optimization report

## Purpose

This report captures the broader startup-latency research done after the earlier remediation session.

It is meant to answer three follow-up questions:

- what could change in the bootstrap mechanisms,
- what could change in the runtime image contents / NixOS config,
- and what else could be optimized beyond those two buckets.

This was a research-focused session against current `main`.
No code changes were made; this report is the only repo addition from the session.

---

## Executive summary

The earlier remediation conclusion still holds after re-verifying current `main`:

- the **dominant first-cold-start cost** is still the runtime image ingestion/materialization path,
- the **dominant later non-running / restart cost** is still repeated bootstrap + lifecycle work,
- and the runtime image is probably carrying substantially more closure than the default workflow needs.

### Highest-value conclusions

1. **If the goal is to reduce the very first cold start, the best next target is still changing the image ingestion path.**
   - Current `main` still uses a flat rootfs tar -> `docker import` flow.
   - That is likely the main first-start bottleneck.

2. **A slimmer default image is the strongest conservative parallel mitigation.**
   - Current `main` bundles several heavyweight tools directly into the base runtime image.
   - The biggest measured suspect is `llm-agents.agent-browser`.

3. **Bootstrap still matters, but mostly after the image already exists locally.**
   - Current bootstrap still recursively repairs ownership and still uses a chatty readiness loop.
   - Those are good targets for later-start improvements, not the primary first-cold-start target.

4. **Current `main` and the remediation handoff doc are not the same state.**
   - The remediation report describes worktree behavior.
   - Current `main` still does unconditional image removal/re-import on the packaged helper path.

## Verified current-main state

## 1. Current `main` still uses flat rootfs import on the interactive path

Current cold `frag enter` behavior on `main` is still:

1. resolve profile + runtime spec,
2. if no matching running container exists:
   - load runtime image,
   - start container,
   - wait for bootstrap,
   - exec user command.

Relevant files:

- `_scripts/frag/src/frag/cli.py`
- `_scripts/frag/src/frag/docker_runtime.py`
- `_scripts/frag/src/frag/image_assets.py`

Current packaged-image helper behavior on `main`:

- `packages/frag/images.nix` still:
  - extracts the NixOS tarball,
  - rewrites `/home/agent` and `/sw`,
  - repacks a rootfs tar,
  - then the helper does:
    - `docker image rm -f "$image_ref" || true`
    - `docker import ... "$runtime_rootfs" "$image_ref"`

So current `main` still forces image materialization directly onto the cold startup path.

### Practical meaning

This strongly supports the existing hypothesis that the first cold start is dominated by image ingestion/materialization, not by small bootstrap tweaks.

## 2. Current `main` and the remediation report describe different states

The remediation handoff report is still useful, but it describes the separate remediation worktree behavior, not the exact current `main` behavior.

Two important distinctions:

- the remediation report says the loader helper no longer blindly removes/re-imports every time,
- but current `main` still does exactly that in `packages/frag/images.nix`.

Also:

- the remediation report references `_scripts/frag/scripts/smoke_verify.py`,
- but that file is not present in current `main`.

### Practical meaning

Future benchmarking and planning should keep three states separate:

- current `main`,
- the remediation worktree,
- and doc-described historical results.

## 3. Measured current-main image size signals point to image materialization as the main first-start problem

Measured during this session:

- current runtime rootfs tar file size: **3,567,206,400 bytes** (~3.4 GiB)
- `nix path-info -Sh` for that rootfs artifact closure: **~6.1G**
- imported Docker image reported size: **1,155,292,234 bytes**
- `docker image history` shows a single imported layer of roughly **3.62GB**
- shared assets size: **~856K**

### Interpretation

- The image is not dominated by prompts or small config payloads.
- The current artifact shape is large enough that disk IO and unpack/import behavior are plausible first-order costs.
- The imported image shape is a single imported layer, not a layered Docker/OCI archive with reusable image layers.

## 4. Current runtime image contents are likely over-broad for the default workflow

Current `packages/frag/runtime-system.nix` includes these directly in the base image:

- `bashInteractive`
- `git`
- `less`
- `llm-agents.agent-browser`
- `llm-agents.code`
- `llm-agents.omp`
- `llm-agents.opencode`
- `mcp-nixos`
- and standard runtime utilities

Measured direct closure suspects from this session:

| Package                    | Approx closure size | Notes                       |
| -------------------------- | ------------------: | --------------------------- |
| `llm-agents.agent-browser` |            ~1.7 GiB | biggest suspect by far      |
| `git`                      |          ~347.7 MiB | meaningful for a base image |
| `llm-agents.omp`           |          ~330.8 MiB | significant                 |
| `mcp-nixos`                |          ~313.5 MiB | significant                 |
| `llm-agents.code`          |          ~296.8 MiB | significant                 |
| `llm-agents.opencode`      |          ~221.3 MiB | significant                 |

### Interpretation

The runtime image does not look large because of many tiny components. It looks large because it includes a few heavy packages directly in the default base image.

## 5. Current bootstrap path still has clear recurring costs

Current `main` bootstrap/runtime behavior still includes several recurring costs after the image already exists:

### Ownership repair

`_scripts/frag/src/frag/bootstrap.py` still recursively `chown`s:

- `state_profile`
- then `state_profile/home` again
- then `/tmp/frag`

Important detail:

- `state_profile/home` is nested under `state_profile`,
- so that subtree is currently traversed twice.

### Readiness waiting

`_scripts/frag/src/frag/docker_runtime.py` still waits for readiness with a 250ms poll loop that may perform:

- `docker exec` readiness/token check,
- `docker exec` bootstrap-status check,
- `docker volume inspect` + host read for persisted failure,
- plus log reads or extra failure handling on timeout / retry paths.

Subagent measurement on this host suggested warm command overhead of roughly **100-110ms per retry iteration** before the 250ms sleep, even before counting actual bootstrap work.

### Container lifecycle

Current cold path still uses `docker run --detach --rm`.
That means:

- stopping the container removes it,
- later re-entry recreates the container,
- bootstrap work repeats for each non-running restart.

### Interpretation

These are good recurring-start optimization targets, but they do not look like the dominant first-cold-start issue.

---

## 6. Current host/runtime context does not point to the worst-known Docker slow path

Verified during this session:

- Docker is running in **rootless** mode
- storage driver is **overlayfs**
- containerd snapshotter is in use
- this is **not** the `vfs` path
- the user service was active during testing
- `Linger=no`

### Interpretation

The worst documented rootless storage-driver pathology is not the current local explanation.

That said, host/runtime conditions still matter and should be recorded during future benchmarks:

- rootless vs rootful,
- storage driver / snapshotter,
- backing filesystem,
- daemon warm vs daemon cold,
- local Docker image-store size.

## Optimization opportunities by area

## A. Bootstrap mechanism changes

These are the most plausible bootstrap-side improvements.
They mainly target repeated non-running starts, restart/re-entry, and startup consistency once the image is already local.

### A1. Backport ownership metadata caching

Why it matters:

- current `main` still does recursive ownership repair every non-running cold path,
- the earlier remediation work already validated this direction as useful for repeat starts.

Expected effect:

- **low** effect on the very first image-cold start,
- **high** effect on later starts where image materialization is already solved or cached.

### A2. Remove duplicate `state_profile/home` traversal

Why it matters:

- current code recursively walks `state_profile`,
- then recursively walks `state_profile/home` again,
- even though `home` is already inside `state_profile`.

Expected effect:

- modest but straightforward recurring-start win,
- likely low risk.

### A3. Replace the multi-probe readiness loop with a single readiness source

Why it matters:

- current readiness is chatty and does multiple Docker round trips per poll,
- it also depends on identity-wrapper state even though readiness fundamentally comes from files under profile state.

Possible direction:

- use one authoritative readiness file / status file / marker contract,
- keep failure detail reporting, but remove extra retry-path probes.

Expected effect:

- moderate recurring-start improvement,
- reduced complexity and fewer cross-process failure modes.

### A4. Shift more one-time setup out of `enter`

Possible direction:

- move some directory/config materialization into profile creation or another earlier lifecycle step,
- leave `enter` to do only what must be done at container creation time.

Expected effect:

- moderate later-start win,
- little effect on first image import.

### A5. Change container lifecycle so exact-match containers can be reused

Current limitation:

- `docker run --detach --rm` means stop implies container destruction.

Possible directions:

- warm-container default reuse,
- or a `docker create` -> `docker start` -> `docker exec` lifecycle for exact-match runtime metadata.

Expected effect:

- potentially **large** improvement for restart / re-entry,
- especially once image loading is no longer repeated.

Main tradeoff:

- more lifecycle/state invalidation logic.

### A6. More radical ownership-model redesign

Possible direction:

- reduce or avoid large recursive ownership repair via a different uid/gid or mount strategy.

Expected effect:

- potentially large,
- but correctness risk is much higher because it touches:
  - user identity,
  - home persistence,
  - writable state,
  - workspace ownership,
  - read-only shared assets.

This is a design study candidate, not a quick optimization.

## B. Runtime image contents / NixOS config changes

These are the strongest conservative options for improving first cold start without immediately replacing the whole runtime model.

### B1. Add a slim default image

The existing catalog/image-key structure already suggests that multiple packaged image entries are feasible.

Practical direction:

- keep a richer `main` / `full` style image for optional workflows,
- make a slimmer default image for the common path.

Expected effect:

- likely one of the best conservative ways to reduce first cold start.

### B2. Make `agent-browser` optional first

Why it matters:

- it is the largest measured closure suspect by far.

If browser tooling is not required for the default workflow, this is the cleanest first candidate to split out of the base image.

Expected effect:

- potentially major image-size reduction.

### B3. Re-evaluate other heavy default packages

Candidates:

- `mcp-nixos`
- `llm-agents.omp`
- `llm-agents.code`
- `llm-agents.opencode`
- possibly `git`

Expected effect:

- meaningful aggregate reduction if several move out of the default image.

Main open question:

- which of these are truly required for the default, core frag workflow?

### B4. Continue externalizing static payloads where appropriate

Current repo already externalizes shared assets and bind-mounts them read-only.
That pattern could be extended carefully.

Possible direction:

- keep more static, read-only payload outside the base image where correctness allows.

Tradeoff:

- more host/package coupling,
- less hermeticity,
- more mount complexity.

### B5. Avoid over-optimizing tiny components before the large ones

The main current opportunity is not swapping a small shell utility.
The main opportunity is removing or splitting the largest direct closure suspects first.

## C. Other optimization avenues

These are broader architectural or operational levers beyond bootstrap-only or image-content-only changes.

### C1. Replace flat rootfs import with a loadable Docker/OCI archive path

This remains the highest-value first-cold-start hypothesis.

Current path:

- build NixOS tarball,
- unpack it,
- rewrite pieces,
- repack flat rootfs tar,
- `docker import` the result.

Most promising next direction:

- preserve runtime invariants if possible,
- but ship a Docker/OCI archive shape that can be `docker load`ed instead.

Why this matters:

- it changes the ingestion path itself,
- it avoids treating the runtime as a generic imported filesystem tar,
- and it opens the door to image-archive / layer semantics that the current path does not use.

Main caveat:

- this is not a trivial builder swap,
- because current frag runtime assumptions rely on NixOS-derived invariants such as `/init`, `/sw`, `/home/agent`, and current user/home behavior.

### C2. Test compressed import artifacts if import must remain temporarily

Current `main` repacks to a large plain tar.
Docker import accepts compressed archives too.

That makes this a good conservative experiment:

- keep the current runtime model,
- compare raw tar import vs compressed tar import.

This is not yet proven to win overall.
It depends on whether the local bottleneck is dominated more by disk IO or by decompression CPU.

### C3. Move image materialization off the user-visible hot path

Possible directions:

- explicit prewarm / prefetch command,
- prewarm keyed by exact packaged identity,
- pre-load during profile creation,
- or another non-interactive preparation phase.

Expected effect:

- high UX value,
- though total work may remain similar.

### C4. Add production phase instrumentation

Current repo status output is useful, but it does not split measured time into phases in the production path.

Recommended phase timing:

- image helper / image load time,
- container create/start time,
- bootstrap wait time,
- first exec time,
- daemon cold vs warm distinction where possible.

Expected effect:

- not a direct speedup,
- but very high leverage for future optimization decisions.

### C5. Add image retention / GC discipline

Measured local Docker state is already large enough that hygiene matters:

- `docker system df` showed very large total image usage,
- many old `frag-main:*` tags remain locally.

This is not the main first-cold-start fix, but it matters for:

- long-term host health,
- disk usage,
- and avoiding pathological local image-store growth.

### C6. Capture daemon-cold and host-variance effects explicitly

During this active session, daemon-cold cost did not look like the primary local explanation.
But it may still matter:

- after reboot,
- after logout,
- on slower disks,
- on different storage-driver setups.

Future benchmarks should record host/runtime context alongside timings.

## Prioritized recommendation stack

## Tier 1 — highest-value next targets

1. **Prototype a Docker/OCI archive load path instead of flat rootfs import.**
2. **Create a slim default image.**
3. **Make `agent-browser` optional first.**
4. **Restore exact-identity local image reuse on current helper/load logic.**

## Tier 2 — strongest repeat-start/runtime wins

5. **Backport ownership caching.**
6. **Remove duplicate `state_profile/home` chown traversal.**
7. **Simplify readiness to one authoritative source.**
8. **Add phase timing to the production path.**

## Tier 3 — broader UX/operational improvements

9. **Add image prewarm / prefetch keyed by packaged identity.**
10. **Explore warm-container reuse or `create/start/exec` lifecycle.**
11. **Add image retention / GC policy for stale frag image tags.**

## Tier 4 — higher-risk design studies

12. **Externalize more runtime payload through read-only mounts.**
13. **Redesign ownership / identity handling to reduce recursive handoff.**

## Recommended concrete experiments

## Experiment 1: archive/load prototype

Goal:

- determine whether a Docker/OCI archive + `docker load` path materially improves first cold start relative to flat rootfs `docker import`.

Success criteria:

- preserves required runtime invariants,
- first-cold-start timing improves materially,
- and correctness remains intact.

## Experiment 2: slim-image prototype

Goal:

- quantify the first-cold-start effect of removing the largest non-essential packages from the default image.

Recommended order:

1. split out `agent-browser`,
2. then evaluate `mcp-nixos`, `omp`, `code`, `opencode`,
3. then re-evaluate `git`.

## Experiment 3: compressed-import benchmark

Goal:

- compare raw rootfs tar import vs compressed rootfs import using the current runtime model.

Why:

- this is a lower-risk experiment than changing the archive format entirely.

## Experiment 4: recurring-start bootstrap optimization prototype

Goal:

- quantify the value of:
  - ownership caching,
  - duplicate-home-traversal removal,
  - simpler readiness contract.

## Experiment 5: lifecycle reuse prototype

Goal:

- compare current `run --rm` behavior against exact-match container reuse approaches.

Focus metric:

- restart / re-entry time after stop.

## Experiment 6: instrumentation pass

Goal:

- attach timings to the real production phases so future sessions do not have to infer the dominant cost indirectly.

## Important open questions

These were not resolved in this session:

1. **Which heavy in-image tools are truly required for the default workflow?**
2. **Can a `docker load` / OCI-style path preserve current frag runtime guarantees cleanly?**
3. **Does compressed import help on this host, or does decompression CPU cancel out the IO win?**
4. **How much hermeticity / portability is acceptable to trade away in exchange for a smaller image?**
5. **How much restart-state complexity is acceptable if warm-container reuse is introduced?**

---

## Final conclusion

This broader research session strengthened the earlier conclusion rather than overturning it.

### Most important takeaway

If the main user-visible problem is that the **first cold start is too slow**, the next serious optimization target should be:

- **changing or bypassing the current image import/materialization path**,
- while preserving the runtime guarantees frag currently depends on.

### Secondary takeaway

If the goal is to improve **later starts, restart/re-entry, and overall runtime smoothness**, the best next targets are:

- image reuse by exact packaged identity,
- ownership-repair reduction,
- readiness-loop simplification,
- and container-lifecycle reuse.

### Practical planning takeaway

Future work should explicitly distinguish:

- current `main`,
- the earlier remediation worktree,
- and historical doc-described measurements,

so that benchmark conclusions are always attached to the correct code state.
