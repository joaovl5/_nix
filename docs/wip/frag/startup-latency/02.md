## Executive summary

The latency remediation work did **not materially improve the very first cold start** when the image has to be loaded/imported fresh.

It **did materially improve repeated non-reuse / restart behavior** by avoiding repeated image re-imports.

### Measured takeaway

Using the same cleanup-safe smoke driver for both branches:

| Metric           | `main` baseline | remediation worktree |                              change |
| ---------------- | --------------: | -------------------: | ----------------------------------: |
| fresh cold start |         ~95.94s |              ~94.09s |  modest improvement (~1.86s, ~1.9%) |
| reuse            |          ~1.62s |               ~1.48s |          small improvement (~0.14s) |
| restart re-entry |        ~106.98s |              ~10.94s | **major improvement** (~96s faster) |

Interpretation:

- the dominant cost for the **first** cold start is still image ingestion/materialization,
- the remediation successfully removed a large repeated cost on later starts by preventing unnecessary re-imports,
- so the next serious optimization target is still the **initial image import path**, not minor bootstrap tweaking.

This matches the research hypothesis.

## Research findings that drove the work

These came from `docs/wip/frag-startup-latency.md` plus static repo analysis and external source review.

### Likely one-time cost

The biggest one-time cold-start cost was assessed to be:

- current NixOS rootfs tarball creation/import path,
- specifically the NixOS tarball -> Docker `import` model.

### Likely recurring cost

The biggest recurring non-reuse costs were assessed to be:

- recursive ownership repair in bootstrap,
- repeated readiness/status polling,
- unconditional image/helper work on later starts.

### Conservative architectural conclusion

The session did **not** assume a safe immediate switch away from the current NixOS-derived rootfs-import approach.

So the remediation plan deliberately focused on:

- reducing repeated bootstrap work,
- making image loading cache-friendly,
- improving cleanup-safe verification,
- cleaning up image build structure,
- but **not** yet replacing the import-based image ingestion strategy.

## What was implemented in the remediation worktree

## 1. Cleanup-safe verification tooling

Added:

- `_scripts/frag/src/frag/verification.py`
- `_scripts/frag/scripts/smoke_verify.py`
- `_scripts/frag/tests/test_verification.py`

Capabilities added:

- namespaced disposable verification profiles/workspaces,
- cleanup harness with LIFO cleanup order,
- tolerant handling for already-missing cleanup targets,
- narrow retry for transient Docker `volume is in use` cleanup race,
- repo-hosted smoke verification entrypoint,
- explicit `--frag-bin` support so smoke checks target a chosen built artifact,
- timing summary output,
- cleanup verification that leaves no stray `frag-verify-*` containers/volumes.

### Important practical result

This directly fixed one earlier operational pain point:

- verification no longer leaves behind a mess of temporary profiles, containers, and volumes when it succeeds.

---

## 2. Bootstrap/runtime recurring-start optimization

Changed:

- `_scripts/frag/src/frag/bootstrap.py`
- `_scripts/frag/src/frag/docker_runtime.py`

Main changes:

- ownership metadata caching,
- explicit ready marker contract,
- stale ready-state invalidation,
- reduced duplicate ownership work,
- more focused readiness polling.

### Important details

The cached ownership fast path needed several follow-up fixes so it remained correct:

- rewritten metadata files on cache hits still needed owner handoff,
- ephemeral cache still needed chown on cache hits,
- recreated managed profile files on cache hits still needed handoff,
- recreated `state_profile/home` needed handoff,
- recreated managed home subdirectories also needed handoff,
- retryable wait path still needed to surface current-token bootstrap failure before sleeping.

Those fixes were all added test-first in the worktree.

### Net effect

This reduced repeated bootstrap churn and made the startup wait loop cleaner.

But based on final timing comparison, these changes were **not** the primary driver of first-cold-start behavior.

---

## 3. Image build / image loading cleanup

Changed:

- `packages/frag/images.nix`
- `packages/frag/runtime-system.nix`
- `_scripts/frag/src/frag/image_assets.py`
- `_scripts/frag/tests/test_image_assets.py`

Main changes:

- `images.nix` refactored toward reusable `snake_case` helper names/functions,
- helper/script logic extracted into named pieces in the `let ... in` block,
- runtime image includes `fish`,
- runtime user `agent` now uses `fish` as default shell,
- loader helper no longer blindly removes/re-imports the tag every time,
- fast path only short-circuits when the existing Docker image matches packaged identity.

### Important correction

A naive fast path was initially unsafe because it trusted **any** existing image with the same tag.

That was fixed by making the helper verify a packaged identity label before reusing the image.

### Net effect

This appears to be the main reason later restart/re-entry improved so dramatically.

---

## 4. Smoke verification expansion

The smoke driver now covers at least:

- `profile new`
- `profile list`
- `enter -- whoami`
- `enter -- pwd`
- workspace write + host ownership check
- `omp --help`
- writable home persistence across restart
- read-only shared asset enforcement
- timing summary output

### Important corrections made during smoke work

Several real issues were found and fixed while getting the smoke run trustworthy:

- `whoami`/`pwd` had to be asserted, not just executed,
- read-only asset enforcement had to prove the failure was actually a read-only/permission failure,
- relative `--frag-bin ./result/bin/frag` had to be resolved before any cwd changes,
- workspace gid check had to allow inherited/setgid workspace groups,
- smoke default image key had to be `main`, not `nixos`,
- cleanup needed a narrow retry for transient `volume is in use` errors.

## One notable debugging trap encountered

A real and surprising issue during verification:

- `treefmt` kept rewriting a multi-exception `except (...)` shape in `bootstrap.py` into invalid Python syntax.

The stable fix was to avoid that syntax shape entirely and rewrite the logic into separate `try` blocks:

- one `try` for file reading,
- one `try` for JSON parsing.

This matters because a future agent may otherwise think there is a mysterious flake/build inconsistency when it is actually a formatter-induced syntax rewrite.

## Comparison and interpretation

### 1. First cold start

- `main`: ~95.94s
- remediation: ~94.09s

This is only a small improvement.

### Conclusion

The implemented optimizations did **not** significantly reduce the first cold-start wall time.

That strongly suggests the original research diagnosis was right:

- the dominant first-start cost is still the initial image load/import/materialization path,
- not the repeated bootstrap work that was optimized here.

### 2. Reuse path

- `main`: ~1.62s
- remediation: ~1.48s

This is a small improvement, but not the big story.

### 3. Restart / non-reuse after stop

- `main`: ~106.98s
- remediation: ~10.94s

This is the big improvement.

### Conclusion

The loader fast path and related image-loading changes dramatically improved later restarts by avoiding repeated re-import behavior.

So the remediation work was **not useless** — it just mostly helped the **repeat-start** path, not the **very first** cold start.

## What this means for the next session

## Most likely next optimization target

If the goal is to reduce the very first cold start, the next session should focus on:

- the image ingestion architecture itself.

That likely means evaluating whether `frag` can preserve the current NixOS-derived runtime guarantees while avoiding the current flat rootfs `docker import` cost.

This was intentionally **not** done in the remediation session.

### Likely next design question

Can we move from:

- current NixOS tarball -> rootfs tar -> `docker import`

to something that preserves the runtime contract but avoids the same first-load penalty?

That needs a dedicated design/implementation session.

## Final conclusion

The remediation session produced **useful, validated improvements**, but mostly for:

- repeated starts,
- restart/re-entry behavior,
- cleanup-safe verification,
- and maintainability of image handling.

It did **not** solve the main user-visible problem of the first cold start being very slow.

So the next serious optimization session should focus on:

- **changing or bypassing the expensive image import/materialization path**,
- while keeping the current NixOS-derived runtime guarantees.
